## Mission

Apply the SDD-116 DEMO pattern verbatim to **D-22 LM Status & Operability**: inline the shared
`demo-mode.{js,css}` helper, add a `demoActive()` short-circuit in `refresh()` that renders badged
sample devices (`DEMO_DEVICES`, obvious `demo/…` placeholder ids, sample p50/p95/p99 latency + KV% +
24h heatmap) with **zero network calls** (no fetch, **no EventSource** in DEMO), and give the chat a
canned `[DEMO]` reply. Same opt-in + always-badged SB-077 reconciliation (operator-confirmed SDD-116).

## Grounded design (no new data)

- `DEMO_DEVICES` is built on the existing `FIXED_DEVICES` topology (CPU0/GPU0/GPU1) with sample model
  slots filled — correctly-shaped for the existing `draw()` / `renderDevice()` / `modelRows()` /
  `historyHtml()` render path (no render change).
- `refresh()` gains a `demoActive()` short-circuit: `draw(DEMO_DEVICES)` + the DEMO banner + the badge,
  no `fetchDevices()`.
- The EventSource is guarded by `!demoActive()` (zero network in DEMO).
- `sendChat()` returns a canned `[DEMO]` reply in DEMO (the loopback POST is never hit).
- A contract-lint guard pins the D-22 DEMO no-network path.

## Goals / Non-goals

- Goal: D-22 is DEMO-capable via the shared helper; badged sample devices; zero network in DEMO.
- Non-goal: no change to D-22 live behaviour; no new data model; no web mutation. Broader panels ("so
  many other") remain the sequenced follow-ups.

## Way forward

- **Stage 0 (this doc)** — SDD-118 + INDEX + mandate E11.M118.
- **Stage 1** — D-22 DEMO (helper + `demoActive()` branch + `DEMO_DEVICES` + canned chat) + lint guard.
- **Stage 2** — full gate + Playwright (DEMO on: badged sample devices + latency + heatmap, zero
  lm-status API calls) + commit/push/draft PR.
